# 📝 Engineering Logbook Entry

## 🎯 Strategic Vision & Objectives

* Driver Autonomy: 
  * Shifting permanently away from third-party vendor code. Moving forward, I am creating custom, ground-up implementations for all peripheral drivers required by this telematics project. This approach guarantees full low-level bus control and ensures a complete understanding of our hardware-level interfaces.
* Target Performance Configuration: 
  * Began implementing a lightweight, highly simplistic ILI9341 display driver targeted for ≤ 30 FPS. Because this weather dashboard processes slow-moving environmental metrics, a high-refresh rendering pipeline is unnecessary. A simpler architecture avoids unneeded memory and timing overhead.

------------------------------
## 🛠️ Refactoring Decisions & Technical Implementation
### 1. Unified SPI Bus Wrapper (DONE)

* Consolidated the separate legacy ILI9341_WriteCommand and ILI9341_WriteData routines into a single private bus writer helper: ILI9341_Write().
* This unified function takes a strict type parameter (ILI9341_WriteType_t) to explicitly dictate whether the transmitted byte represents a command payload or data argument.

### 2. Standardized Display Transaction Strategy

* Mapped out and verified the core physical SPI signaling sequence for the ILI9341:
1. Toggle the Data/Command (D/C) GPIO line LOW (command) or HIGH (data).
   2. Pull the Chip Select (CS) GPIO line LOW to select the display chip.
   3. Transmit the data byte over raw STM32 SPI shift registers.
   4. Return the CS line HIGH to release the display bus.

------------------------------
## 📐 Code Quality & Repository-Wide Standardization
### 1. Clean Magic Number Elimination

* Established a strict zero-tolerance policy for raw hex values or anonymous addresses inside driver source code. Every register coordinate and command configuration parameter must be assigned to an explicit, documented macro inside the .h header file.

### 2. Consistent Code File Geography

* Prototypes: Private static function forward declarations must reside exclusively at the very top of the .c file.
* Implementations: Private helper bodies must sit completely at the bottom of the .c file, keeping the public user-facing API positioned cleanly in the middle for instant skimmability.

### 3. Doxygen Documentation & Visual Noise Reduction

* Public API: Comprehensive Doxygen blocks must sit in the header file directly above the exposed declarations.
* Private Helpers: High-density Doxygen annotations must sit inside the source file, directly above the active function definition.
* Visual Noise Cleanup: Completely stripped out outdated, redundant trailing comment structures and bulky old-school block separators.
* Removed legacy function headers: 
  * `/****************** Name() ******************/`
* Removed trailing block braces: `} /* Name() */`
* Action Item: I must backtrack through the codebase to update all previously implemented drivers (e.g., BME280) to match this modern style constraint.
